# ADR-001: Identity V2

Status: Accepted

Date: 2026-07-21

## Context

APG supports several ways for a person to authenticate, including Email OTP and Telegram Bot login. More providers may be added later, such as VK Web Login and Google.

The platform needs a stable identity model where authentication providers do not create separate user accounts for the same human. Business data such as profile, friends, keys, rewards, messages, bookings, achievements, referrals, and settings must remain attached to one canonical APG user regardless of the login method.

The previous mixed model allowed multiple sources of truth for account identity. That made identity linking, login restore, and provider-specific authentication vulnerable to split-brain behavior.

## Decision

APG Identity V2 is the authoritative identity architecture.

PostgreSQL is the source of truth for APG users, provider identities, and identity links.

Firestore profile data may keep derived or cached identity-facing fields for display compatibility, but those fields are not authoritative and must not override Identity V2.

## Core invariant

One human equals one canonical APG user.

A canonical APG user may have multiple provider identities linked to it:

- Email identity
- Telegram identity
- Future VK identity
- Future Google identity
- Other future provider identities

A provider identity must not silently create a second APG user if it already belongs to an existing canonical user.

## Identity ownership

Each provider identity belongs to exactly one canonical APG user.

The authoritative relationship is:

```text
APG User
  -> Email identity
  -> Telegram identity
  -> Future provider identities
```

Identity links must be unique by provider and provider user id.

Normalized provider identifiers are canonical. For Telegram, numeric ids are canonical and legacy `tg_<id>` aliases are not used as active identities.

## Authentication flow

Authentication providers resolve through Identity V2.

Email login resolves the normalized email identity and returns the canonical APG user.

Telegram login resolves the normalized Telegram identity and returns the canonical APG user.

If an identity does not exist and account creation is allowed for that flow, Identity V2 creates the identity and canonical user transactionally.

If an identity exists, the existing canonical APG user is returned.

## Linking flow

Identity linking attaches a new provider identity to the currently authenticated canonical APG user.

Linking must be completed only after persistent Identity V2 storage confirms the link.

If the provider identity already belongs to the same canonical user, linking is idempotent.

If the provider identity belongs to another canonical user, the operation must fail with an identity conflict. Generic automatic merge is not part of the runtime linking flow.

## Conflict policy

Runtime authentication and linking must not perform speculative merges.

When two identities point to different canonical APG users, the system returns an explicit conflict and preserves both accounts until a controlled production operation is approved.

Controlled merge operations are exceptional maintenance actions. They must preserve business data and must not be implemented as generic runtime behavior.

## Storage policy

PostgreSQL Identity V2 is authoritative for:

- APG users
- Email identities
- Telegram identities
- Future provider identities
- Identity links
- Identity sessions and verification tokens where applicable

Firestore is not an identity source of truth.

The remaining allowed Firestore auth-related runtime collection is `telegramAuthSessions`, which represents temporary Telegram bot session correlation, not permanent identity ownership.

## Derived profile data

Profile fields such as `linkedTelegram` are derived from Identity V2.

During login, bootstrap, restore, profile refresh, and logout transitions, derived profile data must not resurrect stale provider links that are absent from Identity V2.

If a display cache is maintained, it must be synchronized from Identity V2 to the profile layer, never from the profile layer back into Identity V2.

## Session correlation

Telegram login and Telegram linking use application-owned session identifiers.

The required business identifiers are:

- `state`
- `loginSessionId`, when present
- `telegramSessionId`, when present
- client `requestId`, only when explicitly generated by APG frontend and stored as part of the APG contract

Infrastructure request ids, including reverse proxy or platform-generated `x-request-id`, are not business session identifiers and must not be required for Telegram session validation.

## Deployment invariant

Backend production deployments must be immutable and verifiable.

The running backend must expose runtime version metadata and must match the repository commit used for deployment.

Deployments must not rely on mutable `latest` images as the source of truth.

Runtime verification must confirm:

- Git SHA
- App version
- Image digest
- Critical file checksum where required

## Consequences

New authentication providers must integrate with Identity V2 instead of creating provider-specific user models.

UI screens for login methods must read linked provider state from Identity V2-derived data.

Legacy identity aliases may exist only as retired historical artifacts. Active authentication and linking must use normalized provider identifiers.

Production release readiness requires identity integrity checks to pass with no duplicate normalized identities, no orphan identity links, valid canonical users, and no canonical drift.
